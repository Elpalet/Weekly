# 个人周报写作

## 工作流

1. 阅读 [references/writing-style.md](references/writing-style.md)，按其中的固定章节和语气组织内容。
2. 收集本周日期区间、已完成事项、方法/软件/参数、结果或判断、未解决问题、下周动作，以及用户提供的图或链接。信息不足时仅询问影响真实性的关键缺口，不虚构完成度或结果。
3. 先产出三段式正文草稿：`工作内容：`、`问题：`、`下周计划：`。保持研究记录式表达，保留技术名称、数字、阈值和链接。
4. 用户只需要文字时，直接交付可粘贴正文；用户需要 `.docx` 时，再读取 [references/docx-format.md](references/docx-format.md) 并使用 DOCX/文档工具创建文件。
5. 生成 DOCX 时使用文件名 `周报YYYYMMDD-YYYYMMDD.docx`；不得修改历史参考文件。输出前检查三个固定分区、日期范围、术语与计划是否完整。

## 写作约束

- 以事实进展为主，避免企业化总结、宣传式措辞和空泛“成果显著”表述。
- 不强行补写摘要、标题页、风险矩阵或管理型表格；历史周报的核心是短段落、列表与必要结果图。
- 内容较长时，按项目或流程分块，并按“任务/方法 -> 当前结果 -> 判断或下一动作”的顺序铺陈。
- 对报错、受阻、待验证结论如实写入`问题：`，可使用“目前”“暂时”“正在排查”“不确定”等限定语。
- 对用户原始笔记进行轻量整理与明显错别字修正，但不把自然的科研记录语气改写成正式公文。

## DOCX 输出

- 创建或编辑 Word 文件时，同时启用可用的文档/DOCX 技能，遵守其结构检查和渲染验证流程。
- 本 skill 中的个人历史版式要求优先于通用新文档设计预设；不要擅自引入封面、彩色标题或额外页眉页脚。
- 如需参考原件，读取最近或内容规模相近的原始周报，仅用于核对样式和布局，不复制历史工作内容到新报告。
